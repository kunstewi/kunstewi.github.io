What Is Playwright?

Playwright is a modern, open-source end-to-end testing framework built by Microsoft. Released in 2020, it was created by the same team that originally built Puppeteer at Google. Its primary goal is to provide a single API to automate Chromium, Firefox, and WebKit — meaning your test suite runs against all major browser engines without changing a line of test code.

Unlike older tools such as Selenium (which relies on WebDriver, a W3C standard protocol built as a remote HTTP layer), Playwright communicates with browsers via the Chrome DevTools Protocol (CDP) for Chromium and custom protocol implementations for Firefox and WebKit. This gives Playwright deeper, faster, and more reliable browser control.

Why Playwright Over Alternatives?

Feature Playwright Cypress Selenium
Multi-browser (real engines) ✅ Chromium, Firefox, WebKit ⚠️ Chromium only (Firefox beta) ✅ via WebDriver
Auto-waiting ✅ Built-in ✅ Built-in ❌ Manual waits
Parallel test execution ✅ Native ⚠️ Paid plan for parallelism ✅ Grid required

iframes / shadow DOM ✅ First-class ⚠️ Partial ⚠️ Manual
Multiple tabs/windows ✅ ❌ ✅
Mobile emulation ✅ Device profiles ⚠️ Viewport only ⚠️ Limited
Component testing ✅ ✅ ❌
Trace viewer ✅ Built-in ✅ ❌

How Playwright Works Internally

Understanding Playwright's architecture is key to writing tests that are fast, stable, and debuggable.

The Protocol Layer

Playwright maintains its own browser protocol implementations:

  • Chromium: Uses the Chrome DevTools Protocol (CDP) directly over a WebSocket connection.
  • Firefox: Playwright maintains a custom browser patch (juggler) that exposes a CDP-like protocol. It ships its own patched Firefox binary.
  • WebKit: Similarly, Playwright patches WebKit and ships its own binary, exposing automation hooks not available in stock Safari.

This means Playwright does not use WebDriver at all. It speaks directly to the browser's internal messaging bus, which is why it can do things like:

  • Intercept network requests at the protocol level before they hit the network stack.
  • Emulate devices at the browser-engine level (not just viewport resizing).
  • Control service workers, web workers, and browser extensions.

The Node.js ↔ Browser Bridge

Plaintext snippet 
Test (Node.js)
    │
    ▼
Playwright Node API
    │  (JSON messages over WebSocket / pipe)
    ▼
Browser Server Process
    │  (CDP / Playwright Protocol)
    ▼
Browser (Chromium / Firefox / WebKit)
    │
    ▼
Page (JS execution context)

When you call page.click('#btn'), Playwright:

  1. Serializes the command to JSON and sends it over an IPC pipe (or WebSocket for remote browsers).
  2. The browser server dispatches it to the target page via the protocol.
  3. The browser executes the action and returns an ack with any return value or error.
  4. Playwright's Node layer resolves the corresponding Promise.

The Event Loop & Async Nature

Every Playwright API returns a Promise. There is no synchronous mode. This isn't just a stylistic choice — it reflects that every operation crosses a process boundary. Your test runner (Node.js) and the browser are separate processes, and all communication is inherently async.

Execution Contexts

Each page has two execution contexts:

  • Main world: Where page JavaScript runs. Your app code lives here.
  • Isolated world: A sandboxed JS context Playwright injects its automation helpers into. This prevents your app's JavaScript from accidentally (or intentionally) interfering with Playwright's selectors and actions.

When you call page.evaluate(() => document.title), Playwright runs that code in the main world. When it internally queries selectors, it uses the isolated world.

Core Concepts

Browser, Context, and Page

These three objects form Playwright's hierarchy:

Typescript snippet 
// The low-level way (usually managed by fixtures in practice)
const browser = await chromium.launch();
const context = await browser.newContext();
const page = await context.newPage();
  • Browser: Represents a browser process. Expensive to create; usually one per worker.
  • BrowserContext: An isolated browser session — like a fresh incognito window. Has its own cookies, localStorage, cache, and network state. Very cheap to create; you can spin up dozens per test suite.
  • Page: A single tab within a context.

The key insight: contexts give you isolation without launching a new browser. This is why Playwright tests are fast even when running hundreds of them in parallel.

The `test` Object

Typescript snippet 
import { test, expect } from '@playwright/test';

// Basic test
test('my test', async ({ page }) => { ... });

// With title and options
test('slow test', { timeout: 60_000 }, async ({ page }) => { ... });

// Skip / only
test.skip('not ready', async ({ page }) => { ... });
test.only('focus this', async ({ page }) => { ... });

// Group with describe
test.describe('feature X', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('/feature-x');
  });

  test('does Y', async ({ page }) => { ... });
  test('does Z', async ({ page }) => { ... });
});

Locators: The Heart of Playwright

Locators are Playwright's mechanism for finding elements. Unlike raw selectors that query the DOM once and return a static reference, locators are lazy — they re-query the DOM every time an action is performed on them. This is fundamental to Playwright's retry-ability.

Recommended Locators (Resilient to UI Changes)

`getByRole` — The Gold Standard

Typescript snippet 
// Prefer semantic role locators — they test what users actually perceive
await page.getByRole('button', { name: 'Submit' }).click();
await page.getByRole('link', { name: 'Home' }).click();
await page.getByRole('heading', { level: 1 }).textContent();
await page.getByRole('checkbox', { name: 'Accept terms' }).check();
await page.getByRole('textbox', { name: 'Email' }).fill('user@example.com');
await page.getByRole('combobox', { name: 'Country' }).selectOption('India');

getByRole queries by ARIA roles — the same attributes screen readers use. Tests written this way are both accessible-friendly and resilient to CSS changes.

`getByLabel`

Typescript snippet 
// Finds an input associated with a <label>
await page.getByLabel('Password').fill('secret');

`getByPlaceholder`

Typescript snippet 
await page.getByPlaceholder('Search...').fill('playwright');

`getByText`

Typescript snippet 
await page.getByText('Welcome back!').isVisible();
await page.getByText(/logged in as/i).textContent();

`getByTestId`

Typescript snippet 
// Uses data-testid attribute (configurable)
await page.getByTestId('submit-btn').click();

Configure the attribute name in playwright.config.ts:

Typescript snippet 
use: { testIdAttribute: 'data-cy' }  // or any custom attribute

Chaining Locators

Typescript snippet 
// Scope to a specific region
const modal = page.getByRole('dialog');
await modal.getByRole('button', { name: 'Confirm' }).click();

// Filter locators
await page.getByRole('listitem')
  .filter({ hasText: 'Product A' })
  .getByRole('button', { name: 'Add to cart' })
  .click();

Nth Matching

Typescript snippet 
await page.getByRole('row').nth(2);           // 0-indexed
await page.getByRole('row').first();
await page.getByRole('row').last();

CSS & XPath (Use Sparingly)

Typescript snippet 
// CSS selector
await page.locator('div.card > h2').textContent();

// XPath
await page.locator('xpath=//button[text()="Submit"]').click();

// :has-text pseudo-class (Playwright extension)
await page.locator('li:has-text("Item 3")').click();

// :has() — contains a descendant
await page.locator('section:has(h2)').screenshot();

Assertions

Playwright's expect is built on top of Jest's assertion library but adds web-first assertions that automatically retry until the condition is met or a timeout is reached.

Element Assertions

Typescript snippet 
// Visibility
await expect(locator).toBeVisible();
await expect(locator).toBeHidden();

// State
await expect(locator).toBeEnabled();
await expect(locator).toBeDisabled();
await expect(locator).toBeChecked();
await expect(locator).toBeEditable();
await expect(locator).toBeFocused();
await expect(locator).toBeEmpty();   // no text content

// Content
await expect(locator).toHaveText('Exact text');
await expect(locator).toHaveText(/partial match/i);
await expect(locator).toContainText('substring');
await expect(locator).toHaveValue('input value');
await expect(locator).toHaveValues(['a', 'b']);  // multi-select
await expect(locator).toHaveAttribute('href', '/home');
await expect(locator).toHaveClass(/active/);
await expect(locator).toHaveCSS('color', 'rgb(0, 0, 0)');

// Count
await expect(locator).toHaveCount(5);

// Snapshot
await expect(locator).toMatchAriaSnapshot(`
  - heading "Welcome"
  - list:
    - listitem: "Item 1"
    - listitem: "Item 2"
`);

Page Assertions

Typescript snippet 
await expect(page).toHaveTitle('My App');
await expect(page).toHaveTitle(/My App/);
await expect(page).toHaveURL('https://example.com/dashboard');
await expect(page).toHaveURL(/\/dashboard$/);

Soft Assertions

Soft assertions don't stop the test on failure — they collect all failures and report them together at the end:

Typescript snippet 
await expect.soft(page).toHaveTitle('Correct Title');
await expect.soft(locator).toBeVisible();
// Test continues even if above fail, reports all at the end

Negation

Typescript snippet 
await expect(locator).not.toBeVisible();
await expect(page).not.toHaveURL(/error/);

Custom Timeout Per Assertion

Typescript snippet 
await expect(locator).toBeVisible({ timeout: 10_000 });

Auto-Waiting: The Magic Behind Reliability

This is one of Playwright's most important features. Before performing any action, Playwright automatically waits for the element to be ready. It checks a series of "actionability" conditions:

Condition Checked For
Attached to DOM click, fill, hover, etc.
Visible click, fill, hover, etc.
Stable (not animating) click, fill, etc.
Enabled click, fill, select, etc.
Receives events click — not covered by another element
Editable fill, type, clear

This means you almost never need await page.waitForSelector(...) or await page.waitForTimeout(...) (the latter is an anti-pattern and should be avoided in nearly all cases).

How Retry Logic Works Internally

When you call await locator.click(), Playwright:

  1. Resolves the locator to a DOM element by running the selector query in the isolated context.
  2. Checks all actionability conditions. If any fail, it retries after a short interval.
  3. Performs the action once all conditions pass.
  4. If the action causes a navigation, waits for the navigation to complete.
  5. If the total time exceeds actionTimeout (default: 30s), throws a TimeoutError with a detailed message.

Explicit Waits (When You Need Them)

Typescript snippet 
// Wait for an element
await page.waitForSelector('.toast', { state: 'visible' });
await page.waitForSelector('.spinner', { state: 'hidden' });

// Wait for a URL
await page.waitForURL(/\/dashboard/);

// Wait for network
await page.waitForResponse('**/api/users');
await page.waitForRequest('**/api/login');

// Wait for a function to return truthy in the page
await page.waitForFunction(() => document.readyState === 'complete');
await page.waitForFunction(
  count => document.querySelectorAll('li').length === count,
  5  // argument passed to the function
);

// Wait for an event
const [popup] = await Promise.all([
  page.waitForEvent('popup'),
  page.click('#open-popup')
]);

Fixtures & Test Lifecycle

Fixtures are Playwright's dependency injection system. They replace beforeEach/afterEach hooks with composable, reusable setup/teardown logic.

Built-in Fixtures

Typescript snippet 
test('example', async ({
  page,           // A Page in a new context
  context,        // The BrowserContext
  browser,        // The Browser instance
  browserName,    // 'chromium' | 'firefox' | 'webkit'
  request,        // APIRequestContext for HTTP requests
}) => { ... });

Custom Fixtures

Typescript snippet 
// fixtures.ts
import { test as base } from '@playwright/test';

type MyFixtures = {
  loggedInPage: Page;
  todoPage: TodoPage;
};

export const test = base.extend<MyFixtures>({
  // Fixture that logs in before each test
  loggedInPage: async ({ page }, use) => {
    await page.goto('/login');
    await page.getByLabel('Email').fill('admin@example.com');
    await page.getByLabel('Password').fill('password');
    await page.getByRole('button', { name: 'Log in' }).click();
    await page.waitForURL('/dashboard');

    // Hand the fixture to the test
    await use(page);

    // Teardown (optional) — runs after the test
    await page.close();
  },

  // Fixture using another fixture
  todoPage: async ({ loggedInPage }, use) => {
    const todoPage = new TodoPage(loggedInPage);
    await todoPage.goto();
    await use(todoPage);
  },
});

export { expect } from '@playwright/test';
Typescript snippet 
// my.spec.ts
import { test, expect } from './fixtures';

test('can create a todo', async ({ todoPage }) => {
  await todoPage.addItem('Write Playwright tests');
  await expect(todoPage.items).toHaveCount(1);
});

Fixture Scopes

Typescript snippet 
export const test = base.extend({
  // Default: 'test' scope — created/destroyed per test
  myFixture: async ({}, use) => { ... },

  // Worker scope — shared across all tests in a worker process
  sharedDB: [async ({}, use) => {
    const db = await connectDB();
    await use(db);
    await db.disconnect();
  }, { scope: 'worker' }],
});

Hooks

Typescript snippet 
test.describe('suite', () => {
  test.beforeAll(async () => {
    // Runs once before all tests in this describe block
  });

  test.afterAll(async () => {
    // Runs once after all tests

Page Object Model

Page Object Model (POM) is a design pattern that encapsulates page-specific selectors and interactions into reusable classes.

Typescript snippet 
// pages/LoginPage.ts
import { Page, Locator } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly submitButton: Locator;
  readonly errorMessage: Locator;

  constructor(page: Page) {
    this.page = page;
    this.emailInput = page.getByLabel('Email');
    this.passwordInput = page.getByLabel('Password');
    this.submitButton = page.getByRole('button', { name: 'Log in' });
    this.errorMessage = page.getByRole('alert');
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
  }

  async expectError(message: string | RegExp) {
    await expect(this.errorMessage).toContainText(message);
  }
}
Typescript snippet 
// pages/DashboardPage.ts
export class DashboardPage {
  constructor(private page: Page) {}

  async expectWelcome(name: string) {
    await expect(
      this.page.getByRole('heading', { name: `Welcome, ${name}` })
    ).toBeVisible();
  }
}
Typescript snippet 
// tests/login.spec.ts
import { test, expect } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';
import { DashboardPage } from '../pages/DashboardPage';

test('successful login', async ({ page }) => {
  const loginPage = new LoginPage(page);
  const dashboardPage = new DashboardPage(page);

  await loginPage.goto();
  await loginPage.login('alice@example.com', 'password123');
  await dashboardPage.expectWelcome('Alice');
});

test('invalid credentials', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('wrong@example.com', 'bad');
  await loginPage.expectError('Invalid credentials');
});

Network Interception & Mocking

Route Interception

Typescript snippet 
// Abort all image requests
await page.route('**/*.{png,jpg,jpeg,gif,svg}', route => route.abort());

// Mock an API endpoint
await page.route('**/api/users', route => {
  route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify([
      { id: 1, name: 'Alice' },
      { id: 2, name: 'Bob' },
    ]),
  });
});

// Modify a response
await page.route('**/api/products', async route => {
  const response = await route.fetch();  // Fetch the real response
  const json = await response.json();

  // Modify it
  json.products.push({ id: 99, name: 'Injected Product' });

  await route.fulfill({ response, json });
});

// Conditionally intercept
await page.route('**/api/**', async (route, request) => {
  if (request.method() === 'POST') {
    await route.continue({ postData: '{"injected": true}' });
  } else {
    await route.continue();
  }
});

Authentication & State Management

Reusing Signed-In State

The most efficient pattern: authenticate once, save the storage state, and reuse it across all tests.

Typescript snippet 
// global-setup.ts
import { chromium, FullConfig } from '@playwright/test';

async function globalSetup(config: FullConfig) {
  const browser = await chromium.launch();
  const page = await browser.newPage();

  await page.goto('https://example.com/login');
  await page.getByLabel('Email').fill('admin@example.com');
  await page.getByLabel('Password').fill('password');
  await page.getByRole('button', { name: 'Log in' }).click();
  await page.waitForURL('/dashboard');

  // Save cookies + localStorage to a file
  await page.context().storageState({ path: 'auth/user.json' });
  await browser.close();
}

export default globalSetup;
Typescript snippet 
// playwright.config.ts

Browser Contexts: Isolation Without Overhead

A BrowserContext is the key to fast, isolated tests.

What a Context Isolates

  • Cookies — each context has its own cookie jar
  • LocalStorage / SessionStorage — completely separate
  • IndexedDB — isolated
  • Cache — separate cache
  • Network conditions — can emulate offline, throttle per context
  • Permissions — geolocation, camera, notifications
  • HTTP credentials

Creating Contexts With Options

Typescript snippet 
const context = await browser.newContext({
  // Authentication
  httpCredentials: { username: 'user', password: 'pass' },

  // Mobile emulation
  ...devices['iPhone 14'],

  // Viewport
  viewport: { width: 1280, height: 720 },

  // Locale & timezone
  locale: 'fr-FR',
  timezoneId: 'Europe/Paris',

  // Geolocation
  permissions: ['geolocation'],
  geolocation: { latitude: 48.8566, longitude: 2.3522 },

  // Color scheme
  colorScheme: 'dark',

  // Record video
  recordVideo: { dir: './videos' },

  // Proxy
  proxy: { server: 'http://myproxy.com:8080' },
});

Multiple Users in One Test

Typescript snippet 
test('two users chat', async ({ browser }) => {
  const aliceContext = await browser.newContext({ storageState: 'auth/alice.json' });
  const bobContext = await browser.newContext({ storageState: 'auth/bob.json' });

  const alicePage = await aliceContext.newPage();
  const bobPage = await bobContext.newPage();

  await alicePage.goto('/chat');
  await bobPage.goto('/chat');

  await alicePage.getByRole('textbox').fill('Hello Bob!');
  await alicePage.keyboard.press('Enter');

  await expect(bobPage.getByText('Hello Bob!')).toBeVisible();

  await aliceContext.close();
  await bobContext.close();
});

Parallel Execution & Sharding

Parallelism Model

Playwright tests run in worker processes (separate Node.js processes). By default:

  • Tests within a single file run sequentially in one worker.
  • Different test files run in parallel across multiple workers.
Typescript snippet 
// playwright.config.ts
export default defineConfig({
  workers: 4,           // 4 parallel workers (default: CPU count / 2)
  fullyParallel: true,  // Also parallelize within files
});

Forcing Sequential Tests

Typescript snippet 
test.describe.configure({ mode: 'serial' });

test.describe('checkout flow', () => {
  test('add to cart', async ({ page }) => { ... });

Trace Viewer — The Crown Jewel

The Trace Viewer is Playwright's most powerful debugging tool. A trace is a full recording of a test: every action, screenshot at each step, network request/response, console logs, and source code.

Typescript snippet 
// playwright.config.ts
use: {
  trace: 'retain-on-failure',  // 'on', 'off', 'retain-on-failure', 'on-first-retry'
}

Viewing a trace:

Bash snippet 
npx playwright show-trace test-results/my-test/trace.zip

Or upload to trace.playwright.dev in the browser.

The Trace Viewer shows:

  • A timeline of all actions
  • DOM snapshots before/after each action (fully interactive)
  • Network requests with request/response bodies
  • Console output
  • Source code at each step
  • Call stack

Component Testing

Playwright can test individual React, Vue, Svelte, and Solid components in isolation — without a full browser app.

Bash snippet 
npm init playwright@latest -- --ct  # Add component testing
Typescript snippet 
// Button.spec.tsx
import { test, expect } from '@playwright/experimental-ct-react';
import Button from './Button';

test('renders correctly', async ({ mount }) => {
  const component = await mount(<Button label="Click me" />);
  await expect(component).toContainText('Click me');
});

test('emits click event', async ({ mount }) => {
  let clicked = false;
  const component = await mount(
    <Button label="Click me" onClick={() => { clicked = true; }} />
  );
  await component.click();
  expect(clicked).toBe(true);
});

test('disabled state', async ({ mount }) => {
  const component = await mount(<Button label="Disabled" disabled />);
  await expect(component).toBeDisabled();
});

Under the hood, Playwright CT:

  1. Spins up a lightweight Vite dev server.
  2. Mounts the component into a special test page.
  3. Connects Playwright's protocol to that page.
  4. Runs assertions against the live-rendered component.

API Testing

Playwright includes a full HTTP client for API testing, sharing the same assertion library as E2E tests.

Typescript snippet 
import { test, expect } from '@playwright/test';

test('CRUD user API', async ({ request }) => {
  // Create
  const createResponse = await request.post('/api/users', {
    data: { name: 'Alice', email: 'alice@example.com' },
  });
  expect(createResponse.ok()).toBeTruthy();
  const user = await createResponse.json();
  expect(user.id).toBeDefined();

  // Read
  const getResponse = await request.get(`/api/users/${user.id}`);
  await expect(getResponse).toBeOK();
  const fetched = await getResponse.json();
  expect(fetched.name).toBe('Alice');

  // Update
  await request.patch(`/api/users/${user.id}`, {
    data: { name: 'Alice Updated' },
  });

  // Delete
  await request.delete(`/api/users/${user.id}`);
  const deletedResponse = await request.get(`/api/users/${user.id}`);
  expect(deletedResponse.status()).toBe(404);
});

Combining API and UI Tests

A powerful pattern: use the API to set up test state, then test the UI:

Typescript snippet 
test('user sees their orders', async ({ page, request }) => {
  // Set up data via API (fast, reliable)
  const order = await request.post('/api/orders', {
    data: { product: 'Widget', qty: 3 },
  }).then(r => r.json());

  // Test the UI behavior
  await page.goto('/orders');
  await expect(page.getByText(`Order #${order.id}`)).toBeVisible();
});

Visual Comparisons

Playwright has a built-in visual regression testing system using screenshot diffing.

Typescript snippet 
// Captures a screenshot and compares to a stored "golden" image
await expect(page).toHaveScreenshot('homepage.png');
await expect(page.getByRole('dialog')).toHaveScreenshot('modal.png');

On first run, Playwright saves the baseline screenshot. On subsequent runs, it diffs against it and fails if pixels differ beyond a threshold.

Typescript snippet 
// Configure tolerance
await expect(page).toHaveScreenshot('chart.png', {
  maxDiffPixels: 100,
  maxDiffPixelRatio: 0.01,  // 1% of pixels may differ
  threshold: 0.2,           // Color distance threshold per pixel
});

Update golden screenshots:

Bash snippet 
npx playwright test --update-snapshots

Codegen: Record & Replay

Playwright's code generator watches your browser interactions and generates test code automatically.

Bash snippet 
npx playwright codegen https://example.com

This opens a browser with a recording toolbar. As you interact:

  • Clicks, fills, and navigations are translated to Playwright API calls in real-time.
  • It automatically picks the best available locator (getByRole, getByLabel, etc.).
  • Code is shown in a side panel and can be copied.
Bash snippet 
# Generate with specific browser
npx playwright codegen --browser=firefox https://example.com

# Save directly to file
npx playwright codegen --output=tests/recorded.spec.ts https://example.com

Codegen is excellent for bootstrapping tests — but the generated code should always be reviewed and refactored, especially around locators and assertions.

Advanced Patterns & Tips

Handling Dialogs

Typescript snippet 
// Auto-dismiss all dialogs
page.on('dialog', dialog => dialog.accept());

// Handle specific dialog
page.once('dialog', async dialog => {
  expect(dialog.message()).toContain('Are you sure?');
  await dialog.accept('confirmed');  // Pass value for prompt dialogs
});
await page.getByRole('button', { name: 'Delete' }).click();

Handling Popups & New Tabs

  page.waitForEvent('popup'),
  page.getByRole('link', { name: 'Open in new tab' }).click(),

await newPage.waitForLoadState();
await expect(newPage).toHaveURL(/new-page/);

Shadow DOM

Playwright's locators pierce shadow DOM by default — no special handling needed:

Evaluating JavaScript in the Page

Typescript snippet 
// Run JS in main world, return serializable value
const title = await page.evaluate(() => document.title);

// Pass arguments to the page function
const count = await page.evaluate(selector => {
  return document.querySelectorAll(selector).length;
}, '.card');

// Evaluate on a specific element (via ElementHandle)
const text = await page.locator('h1').evaluate(el => el.textContent);

// evaluateHandle returns a JSHandle (for non-serializable objects)
const element = await page.evaluateHandle(() => document.body);

Intercepting Console & Errors

Typescript snippet 
const errors: string[] = [];

  if (msg.type() === 'error') errors.push(msg.text());
});
page.on('pageerror', err => errors.push(err.message));

await page.goto('/');
expect(errors).toHaveLength(0);

TestInfo — Accessing Test Metadata Inside Tests

Typescript snippet

  console.log(testInfo.title);        // Test title
  console.log(testInfo.titlePath);    // ['describe', 'test title']
  console.log(testInfo.project.name); // 'chromium'
  console.log(testInfo.retry);        // 0, 1, 2...

  // Attach files to the report

    body: await page.screenshot(),
    contentType: 'image/png',
  });

  // Add to output path
  const outputPath = testInfo.outputPath('data.json');
});

Parallelism Anti-Patterns to Avoid

  1. Shared mutable state — don't share a single page or global variable across tests that run in parallel.
  2. Depending on test order — each test should be able to run independently unless explicitly serialized.
  3. Hard-coded port conflicts — use webServer.port or dynamic ports in CI.
  4. Sleeping — await page.waitForTimeout(2000) is nearly always wrong. Use proper waits or assertions.
